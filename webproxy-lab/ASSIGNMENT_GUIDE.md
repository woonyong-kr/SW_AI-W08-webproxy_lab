# Web Proxy Lab 과제 정리

이 문서는 현재 저장소 기준으로 이 과제에서 무엇을 구현해야 하는지, 어떤 순서로 공부하면 되는지 빠르게 파악할 수 있도록 정리한 안내서입니다.

## 1. 이 과제에서 풀어야 하는 것

이 과제의 핵심은 HTTP 요청이 네트워크를 통해 어떻게 오가고, 서버가 여러 요청을 어떻게 처리하며, 캐시를 통해 응답 속도를 어떻게 개선하는지 직접 구현해 보는 것입니다.

현재 저장소를 보면 다음 두 파일이 핵심입니다.

- `proxy.c`
  - 프록시 서버의 핵심 구현 파일입니다.
  - 현재는 거의 비어 있으므로 사실상 처음부터 구현해야 합니다.
- `tiny/tiny.c`
  - 원 서버 역할을 하는 Tiny 웹 서버 구현 파일입니다.
  - 현재 저장소 기준으로는 이 파일도 미완성 상태이므로, 로컬 테스트를 위해 먼저 동작하도록 만들어야 합니다.

즉, 이 저장소 기준으로는 단순히 프록시만 채우는 과제가 아니라 다음 흐름 전체를 이해하고 구현해야 합니다.

1. Tiny 서버가 정적/동적 콘텐츠를 제공한다.
2. Proxy 서버가 클라이언트 요청을 받아 원 서버에 전달한다.
3. Proxy 서버가 여러 요청을 동시에 처리한다.
4. Proxy 서버가 작은 응답 객체를 캐시에 저장하고 재사용한다.

## 2. 자동 채점 기준

`webproxy-lab/driver.sh`를 보면 총점은 70점입니다.

- 기본 기능: 40점
- 동시성: 15점
- 캐시: 15점

각 항목은 아래처럼 검증됩니다.

### 기본 기능

프록시를 거쳐 받아온 파일과 Tiny에서 직접 받아온 파일이 같은지 비교합니다.

검사 대상 예시는 다음과 같습니다.

- `home.html`
- `csapp.c`
- `tiny.c`
- `godzilla.jpg`
- `tiny`

즉, 텍스트와 바이너리 파일 모두 정확히 중계해야 합니다.

### 동시성

응답하지 않는 `nop-server.py` 요청 하나가 걸려 있어도, 동시에 들어온 다른 정상 요청을 계속 처리할 수 있어야 합니다.

즉, 순차 처리 프록시는 이 테스트를 통과하지 못합니다.

### 캐시

먼저 프록시를 통해 여러 파일을 받아 캐시에 넣고, 이후 Tiny 서버를 종료한 뒤에도 프록시가 캐시된 파일을 다시 돌려줄 수 있어야 합니다.

즉, 원 서버가 죽은 뒤에도 캐시된 객체는 살아 있어야 합니다.

## 3. 파일별 역할

- `webproxy-lab/proxy.c`

- `webproxy-lab/csapp.c`, `webproxy-lab/csapp.h`
  - CS:APP 스타일의 소켓, 스레드, 세마포어, Robust I/O 래퍼

  - Tiny 정적 콘텐츠 테스트 파일

  - 동적 콘텐츠 예제 CGI 프로그램
- `webproxy-lab/driver.sh`
  - 자동 채점 스크립트

  - 동시성 테스트용 차단 서버
- `webproxy-lab/free-port.sh`
  - 사용 가능한 포트 찾기

## 4. 권장 학습 순서

이 과제는 한 번에 프록시 전체를 짜려 하면 막히기 쉽습니다. 아래 순서로 가면 훨씬 수월합니다.

### 1단계: HTTP와 소켓 흐름 이해

먼저 아래 개념을 짧게라도 정리하고 시작하는 것이 좋습니다.

- TCP 연결 생성과 종료
- 서버의 `listen`, `accept`
- 클라이언트의 `connect`
- HTTP 요청 줄과 헤더 구조
- HTTP 응답 줄, 헤더, 본문 구조
- 텍스트 응답과 바이너리 응답 처리 차이

이 단계에서 목표는 "브라우저 요청이 서버까지 어떻게 가는지"를 머릿속에 그릴 수 있게 되는 것입니다.

### 2단계: Tiny 서버 먼저 완성하기

현재 저장소에서는 `tiny/tiny.c`도 미완성이라, 프록시를 테스트하려면 Tiny가 먼저 동작해야 합니다.

Tiny에서 우선 이해해야 할 함수는 다음과 같습니다.

- `main`
- `doit`
- `read_requesthdrs`
- `parse_uri`
- `serve_static`
- `get_filetype`
- `serve_dynamic`
- `clienterror`

Tiny를 먼저 구현하면 다음을 자연스럽게 배우게 됩니다.

- 요청 줄 파싱
- 헤더 읽기
- 정적 파일 전송
- CGI 실행 흐름
- 에러 응답 생성

즉, Tiny는 프록시를 구현하기 전의 예제 서버라고 생각하면 됩니다.

### 3단계: 순차 프록시 구현

Proxy 구현은 처음부터 동시성과 캐시까지 넣지 말고, 먼저 "한 요청을 정확하게 중계하는 프록시"를 만드는 것이 좋습니다.

이 단계에서 해야 할 일은 다음과 같습니다.

- 클라이언트 연결 수락
- 요청 줄 읽기
- URI에서 호스트, 포트, 경로 분리
- 원 서버에 연결
- 원 서버로 새 HTTP 요청 생성
- 필요한 헤더를 정리해서 전달
- 원 서버 응답을 읽어 클라이언트로 그대로 전달

특히 헤더 처리가 중요합니다. 보통 다음 헤더는 직접 맞춰서 보내는 방식으로 구현합니다.

- `Host`
- `User-Agent`
- `Connection: close`
- `Proxy-Connection: close`

또한 클라이언트가 보낸 나머지 헤더는 필요한 것만 유지하고, 충돌하는 헤더는 직접 통제하는 방식이 안정적입니다.

### 4단계: 동시성 추가

기본 중계가 되면 그 다음에 동시성을 넣습니다.

가장 흔한 방식은 연결마다 스레드 하나를 생성하는 방식입니다.

이 단계에서 중요한 포인트는 다음과 같습니다.

- `accept`한 연결을 각각 별도 스레드에서 처리
- 메인 스레드는 계속 다음 연결을 받기
- 작업이 끝난 스레드는 `detach` 처리
- 공유 자원이 생기면 반드시 동기화 고려

동시성 테스트의 핵심은 "느린 요청 하나 때문에 전체 프록시가 멈추지 않는가"입니다.

### 5단계: 캐시 추가

마지막으로 작은 응답 객체를 메모리에 저장하는 캐시를 구현합니다.

이 단계에서 설계해야 할 것은 다음과 같습니다.

- 객체를 어떤 구조체로 저장할지
- 최대 객체 크기 `MAX_OBJECT_SIZE`를 넘는 응답은 저장하지 않기
- 전체 캐시 크기 `MAX_CACHE_SIZE`를 넘지 않게 관리하기
- 교체 정책을 무엇으로 할지

보통은 LRU 방식으로 구현하는 경우가 많습니다.

캐시를 넣을 때 가장 자주 실수하는 부분은 다음과 같습니다.

- 응답을 끝까지 받기 전에 저장해 버리는 경우
- 스택 버퍼 주소를 그대로 저장하는 경우
- 여러 스레드가 동시에 캐시를 읽고 쓰는데 락이 없는 경우
- 객체 크기 계산을 잘못해서 메모리를 넘기는 경우

## 5. 구현 체크리스트

### Tiny 서버 체크리스트

- 포트를 받아서 `listen` 소켓 생성
- 반복적으로 `accept`
- GET 요청만 처리
- 요청 헤더 읽기
- 정적 콘텐츠 제공
- 동적 콘텐츠 제공
- 오류 상황에서 HTTP 에러 응답 전송

### Proxy 서버 체크리스트

- 포트를 받아서 `listen` 소켓 생성
- 클라이언트 요청 줄 읽기
- 절대 URI 파싱
- 원 서버에 연결
- 새 요청 헤더 조립
- 응답 본문을 바이트 단위로 정확히 전달
- 동시성 처리
- 캐시 저장과 조회

## 6. 추천 학습 방법

가장 효율적인 방법은 "읽고 바로 확인하는 작은 루프"를 반복하는 것입니다.

### 추천 루프

1. `tiny/tiny.c`의 한 함수만 이해한다.
2. 해당 함수가 처리하는 HTTP 흐름을 말로 정리한다.
3. 직접 구현한다.
4. `curl`로 바로 확인한다.
5. Tiny가 되면 같은 방식으로 `proxy.c`를 진행한다.
6. 마지막에 `driver.sh`로 전체 검증한다.

### 공부할 때 집중해야 할 개념

- 파일 디스크립터
- 소켓 프로그래밍
- Robust I/O (`rio_*`)
- HTTP/1.0 요청/응답 형식
- 스레드 기반 동시성
- 임계구역과 락
- 캐시 교체 정책

### 막히지 않기 위한 팁

- 한 번에 전체 구현하지 말고 단계별로 통과시키기
- 먼저 텍스트 파일로 테스트하고, 이후 이미지 같은 바이너리 파일 확인하기
- 헤더 문자열 끝의 `\r\n`을 항상 의식하기
- `printf` 디버깅으로 요청 줄, 헤더, URI 파싱 결과를 바로 확인하기

## 7. 실행과 검증 방법

아래 명령으로 작업하면 됩니다.

```bash
cd webproxy-lab
make
cd tiny && make && cd ..
```

포트가 필요하면:

```bash
./free-port.sh
```

Tiny와 Proxy를 수동 실행해 보려면:

```bash
cd webproxy-lab
./tiny/tiny 8000
./proxy 4500
```

다른 터미널에서 요청 테스트:

```bash
curl http://localhost:8000/home.html
curl --proxy http://localhost:4500 http://localhost:8000/home.html
```

전체 자동 채점:

```bash
cd webproxy-lab
./driver.sh
```

주의할 점은 루트 `README.md`에 `mdriver`라는 표현이 보이더라도, 현재 저장소에서 실제 자동 채점 스크립트는 `driver.sh`라는 점입니다.

## 8. 자주 막히는 포인트

### 1. URI 파싱이 헷갈림

프록시는 보통 절대 URI를 받습니다.

예:

```text
GET http://localhost:8000/home.html HTTP/1.0
```

여기서 다음을 분리해야 합니다.

- 호스트: `localhost`
- 포트: `8000`
- 경로: `/home.html`

### 2. 헤더를 그대로 넘기면 실패함

프록시는 일부 헤더를 그대로 복사하지 말고 직접 통제해야 합니다. 특히 연결 관련 헤더는 명시적으로 정리해야 동작이 안정적입니다.

### 3. 바이너리 파일이 깨짐

이미지 파일은 문자열 함수처럼 다루면 안 됩니다. 읽은 바이트 수만큼 정확히 써야 합니다.

### 4. 동시성을 너무 늦게 고려함

순차 구현이 끝나기 전에는 스레드를 넣지 않는 것이 좋지만, 순차 구현이 끝난 뒤에는 빨리 동시성 구조를 넣어야 합니다. 그렇지 않으면 캐시 구현까지 꼬이기 쉽습니다.

### 5. 캐시와 동기화를 같이 생각하지 않음

스레드가 여러 개면 캐시는 공유 자원입니다. 읽기와 쓰기를 구분해서 보호할지, 단순한 전역 락으로 시작할지 먼저 결정하고 구현하는 것이 좋습니다.
